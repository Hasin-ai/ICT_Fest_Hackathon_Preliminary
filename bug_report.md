# Bug Report - CoWork API

This document provides a detailed breakdown of the bugs identified, analyzed, and fixed in the CoWork codebase during the challenge.

---

### Bug 1: Error Response Code Formats (Spaces vs. Underscores)
- **Location**: Raised across various files, handled in [app/errors.py](file:///Users/bs00851/Documents/Hackathon/ICT_Fest_Hackathon_Preliminary/app/errors.py)
- **Problem**: Error codes generated by the application used underscores (e.g., `ROOM_CONFLICT`, `USERNAME_TAKEN`), whereas the API specification requires them to be returned with spaces (e.g., `ROOM CONFLICT`, `USERNAME TAKEN`).
- **Root Cause**: The developer defined error codes in standard pythonic snake_case, ignoring the specific string requirements in Section 5 of the problem statement.
- **Impact**: Any black-box test asserting the exact value of the `code` field in an error JSON response would fail.
- **Fix**: Modified `app_error_handler` to dynamically replace all underscores (`_`) with spaces (` `) in the returned error code before serialization.
- **Why fix is correct**: It safely maps all raised error codes (such as `ROOM_CONFLICT` to `ROOM CONFLICT`, `USERNAME_TAKEN` to `USERNAME TAKEN`) at the middleware/exception-handler level, keeping individual endpoints clean and standardizing formatting.
- **Potential regressions**: None. All internal app errors now align with the specification.

### Bug 2: Missing Validation for Room Capacity and Rates
- **Location**: [app/schemas.py](file:///Users/bs00851/Documents/Hackathon/ICT_Fest_Hackathon_Preliminary/app/schemas.py#L21-L25)
- **Problem**: Rooms could be created with negative, zero, or empty capacity and hourly rate values.
- **Root Cause**: Lack of field validations on `RoomCreateRequest` fields.
- **Impact**: Room configurations could store nonsensical negative values, leading to database corruption and incorrect booking price calculations.
- **Fix**: Added Pydantic `Field(gt=0)` constraints to both `capacity` and `hourly_rate_cents` in `RoomCreateRequest`.
- **Why fix is correct**: Rejects invalid requests with standard 422 validation errors before they reach the database transaction phase.
- **Potential regressions**: Valid requests with positive integers are unaffected.

### Bug 3: Incorrect UTC Format Designator in Datetime Serialization
- **Location**: [app/timeutils.py](file:///Users/bs00851/Documents/Hackathon/ICT_Fest_Hackathon_Preliminary/app/timeutils.py#L19-L26)
- **Problem**: Stored datetimes returned in JSON responses used the `+00:00` offset suffix instead of the explicit UTC designator `Z`.
- **Root Cause**: Calling `.isoformat()` on timezone-aware UTC datetime objects in Python outputs the `+00:00` offset representation.
- **Impact**: Strict JSON/ISO-8601 validation checks expecting `Z` would fail.
- **Fix**: Updated `iso_utc()` to format output values by replacing `+00:00` with `Z`.
- **Why fix is correct**: It ensures standard ISO 8601 output strings using `Z` while keeping the internal timezone conversion intact.
- **Potential regressions**: None.

### Bug 4: Concurrency Race Condition in Registration
- **Location**: [app/routers/auth.py](file:///Users/bs00851/Documents/Hackathon/ICT_Fest_Hackathon_Preliminary/app/routers/auth.py#L27-L35)
- **Problem**: Concurrent registration calls for a new organization could cause one to fail with a false 409 `USERNAME TAKEN` instead of registering the user as a `member` of the newly created organization.
- **Root Cause**: The organization existence check and organization creation logic are not serialized. Two threads checking for the same org name concurrently will both see `None`, try to insert it, and one will fail the unique database constraint on `Organization.name`, rolling back the entire registration transaction.
- **Impact**: Valid signups fail and receive wrong error response codes under concurrent loads.
- **Fix**: Added `db.execute(text("BEGIN IMMEDIATE"))` to serialize registration transactions.
- **Why fix is correct**: Immediacy on transaction locks prevents any other write transaction from executing concurrently, forcing thread sequential checking/creation.
- **Potential regressions**: A slight serialization delay for simultaneous registration requests, which is desirable and correct behavior.

### Bug 5: Concurrency Race Conditions (TOCTOU) in Booking Creation/Cancellation
- **Location**: [app/routers/bookings.py](file:///Users/bs00851/Documents/Hackathon/ICT_Fest_Hackathon_Preliminary/app/routers/bookings.py#L85-L145)
- **Problem**: Concurrent booking requests could result in double-booking conflicts, quota limits bypass, or concurrent cancellation duplicates.
- **Root Cause**: Verification checks (e.g. `_has_conflict` and `_check_quota`) are performed before the write, but the lock guarding them (`_booking_write_lock`) is only an in-memory lock. Under multiple processes or SQLite concurrent connections, transactions are `DEFERRED` by default, allowing concurrent reads to pass validation checks and commit conflicting inserts.
- **Impact**: Data inconsistency, double bookings, bypassed limits, and unhandled SQLite lock errors (500) under parallel stress tests.
- **Fix**: Executed `db.execute(text("BEGIN IMMEDIATE"))` at the start of write transactions inside `create_booking` and `cancel_booking`.
- **Why fix is correct**: Forces SQLite to acquire a reserved write lock immediately on transaction start, serializing concurrent writes at the database layer.
- **Potential regressions**: None. Concurrency is handled correctly.

### Bug 6: Incomplete and Stale Caching Integration
- **Location**: [app/routers/rooms.py](file:///Users/bs00851/Documents/Hackathon/ICT_Fest_Hackathon_Preliminary/app/routers/rooms.py#L59-L94) and [app/routers/admin.py](file:///Users/bs00851/Documents/Hackathon/ICT_Fest_Hackathon_Preliminary/app/routers/admin.py#L17-L56)
- **Problem**: Cache helpers defined in `app/cache.py` were never utilized in the GET routes, and the usage report cache was not invalidated when a new room was created.
- **Root Cause**: Caching functions were defined but never called in read endpoints. Also, room creation changes the usage report structure but failed to invalidate the report cache.
- **Impact**: The read routes suffered from N+1 query performance degradation, and creating a room while report cache was active left it stale.
- **Fix**:
  1. Integrated `get_availability`/`set_availability` cache checks in the rooms availability endpoint.
  2. Integrated `get_report`/`set_report` cache checks in the admin usage report endpoint.
  3. Added `cache.invalidate_report(admin.org_id)` in the `create_room` endpoint.
- **Why fix is correct**: Integrates the cache safely, and invalidates report caches whenever a booking or a room changes.
- **Potential regressions**: None.

### Bug 7: Export CSV Header Spacing Deviations
- **Location**: [app/services/export.py](file:///Users/bs00851/Documents/Hackathon/ICT_Fest_Hackathon_Preliminary/app/services/export.py#L11-L20)
- **Problem**: Export column headers used underscores (e.g., `reference_code`), but the specification requires exact column names: `"id", "reference code", "room_id", "user id", "start time", "end time", "status", "price cents"`.
- **Root Cause**: Database snake_case column names were written directly into the export headers.
- **Impact**: Grading assertions checking for exact CSV formats would fail.
- **Fix**: Aligned the elements in `EXPORT_HEADER` to match the specification header requirements.
- **Why fix is correct**: Matches the exact string format expected by downstream report parsers.
- **Potential regressions**: None.

### Bug 8: Missing Date Range Validation in Usage Report
- **Location**: [app/routers/admin.py](file:///Users/bs00851/Documents/Hackathon/ICT_Fest_Hackathon_Preliminary/app/routers/admin.py#L17-L29)
- **Problem**: A query to `/admin/usage-report` with a start date greater than the end date (e.g. `from=2026-07-10&to=2026-07-09`) would run silently instead of returning an error.
- **Root Cause**: Lack of date range logical validation.
- **Impact**: Empty reports returned for invalid date range inputs.
- **Fix**: Added logical validation checking if `from_date > to_date` and raising `AppError(400, "INVALID_BOOKING_WINDOW", ...)` accordingly.
- **Why fix is correct**: Rejects invalid query params before processing.
- **Potential regressions**: None.

### Bug 9: N+1 Database Query Performance in Usage Report
- **Location**: [app/routers/admin.py](file:///Users/bs00851/Documents/Hackathon/ICT_Fest_Hackathon_Preliminary/app/routers/admin.py#L41-L65)
- **Problem**: The `/admin/usage-report` endpoint suffered from $O(N)$ database query iterations. It loaded all rooms, then queried the `bookings` table individually for each room to count and sum the bookings.
- **Root Cause**: Implementing an ad-hoc loop query pattern rather than using SQL aggregation and joins.
- **Impact**: Significant response latency and resource consumption under load for organizations with large numbers of bookable rooms.
- **Fix**: Replaced the loop query with a single optimized query utilizing `outerjoin`, `group_by`, and `func` aggregation (`count` and `sum`) to fetch all room stats in one SQL statement.
- **Why fix is correct**: Retains the exact same response layout and business logic (including zero-booking rooms) while reducing the database load to exactly 1 query.
- **Potential regressions**: None.

### Bug 10: SQLite Default DELETE Journal Mode (Concurrency Bottleneck)
- **Location**: [app/database.py](file:///Users/bs00851/Documents/Hackathon/ICT_Fest_Hackathon_Preliminary/app/database.py#L13-L17)
- **Problem**: SQLite connections defaulted to the standard `DELETE` journal mode, which acquires exclusive locks on writes and blocks concurrent reads or parallel writes.
- **Root Cause**: The connection pool did not configure concurrent SQLite tuning parameters.
- **Impact**: High-frequency concurrent booking creations or cancellations could result in thread starvation, database lock timeouts, or application liveness (hang) warnings.
- **Fix**: Added connection listeners to execute `PRAGMA journal_mode=WAL` (Write-Ahead Logging) and `PRAGMA synchronous=NORMAL`.
- **Why fix is correct**: Allows concurrent readers and writers to run safely in parallel without blocking one another, significantly improving concurrent write throughput.
- **Potential regressions**: None.
